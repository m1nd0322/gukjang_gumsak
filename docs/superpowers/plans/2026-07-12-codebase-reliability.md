# 코드베이스 안정화 및 일일 리포트 복구 계획

> 작성일: 2026-07-12
>
> 범위: 고장 난 FnGuide 수집 경로, 백테스트 회계, Flask API 경계, DuckDB 캐시, 문서와 CI

## 목표와 완료 기준

- 폐쇄된 FnGuide HTML 페이지에 의존하지 않고 세 가지 스크리닝 데이터가 다시 수집된다.
- 일일 리포트가 브라우저/Chrome 없이 동일한 점수 계산과 백테스트 단계로 진행된다.
- 거래 비용과 여러 매수분의 매도 손익이 현금 흐름과 일치한다.
- 잘못된 API 입력과 중복 실행 요청은 500 오류나 중복 작업을 만들지 않는다.
- 캐시가 충분한 경우 불필요한 KRX 지수 요청을 하지 않는다.
- KRX 로그인 정보가 없는 새 환경에서도 저장소 매핑과 yfinance 대체 경로로 웹 백테스트가 실행된다.
- 회귀 테스트, 정적 검사, 컴파일, 실데이터 스모크 테스트가 모두 통과한다.
- README와 GitHub Actions가 실제 설치·실행 방식과 일치한다.

## 1. 회귀 테스트로 결함 고정

대상 파일:

- `tests/test_screening.py`
- `tests/test_backtester.py`
- `tests/test_app.py`
- `tests/test_daily_report.py`
- `tests/test_stock_db.py`

검증할 동작:

1. UTF-8 BOM이 포함된 FnGuide JSON을 정상 해석하고 기존 한국어 컬럼으로 매핑한다.
2. 국민연금 행은 요청 종목과 실제 페이지 종목코드가 일치할 때만 채택한다.
3. 세 데이터셋의 종합 점수와 상세 컬럼은 기존 규칙을 유지한다.
4. 최초 거래 비용은 전체 수익률과 MDD에 반영된다.
5. 여러 차례 매수한 수량을 매도하면 각 매수 로트가 FIFO로 정확히 청산되고 실현손익 합계가 현금 흐름과 일치한다.
6. Flask DB 조회는 잘못된 테이블·페이지 인수를 4xx로 반환하고, 갱신/백테스트는 상태를 스레드 시작 전에 예약한다.
7. DuckDB 티커 캐시의 최신 갱신 시각을 기준으로 신선도를 판단하고, 완전한 지수 범위는 재조회하지 않는다.

## 2. 스크리닝 데이터 계층 복구

대상 파일:

- 새 파일 `screening.py`
- `app.py`
- `daily_report.py`
- `stock_screener.py`

구현:

1. 턴어라운드와 수급 데이터는 현재 제공되는 FnGuide JSON 피드를 `requests`로 읽는다.
2. 응답 상태뿐 아니라 JSON 구조를 검증해 HTTP 200 오류 페이지도 실패로 식별한다.
3. 국민연금 데이터는 `ticker_map.json`의 종목별 FnGuide Snapshot 페이지에서 `국민연금공단` 5% 공시 행을 제한된 병렬 요청으로 수집한다.
4. 우선주가 보통주 페이지로 리다이렉트되는 경우 제목의 실제 종목코드로 교차 검증해 중복을 제거한다.
5. 데이터 수집과 점수 계산을 `screening.py`로 모아 웹 앱, 정적 리포트, 일일 리포트가 같은 구현을 사용하게 한다.
6. 개별 소스 오류는 기본 호출에서 기록하고 나머지 소스를 유지하되, 웹·정적·자동 리포트는 엄격 모드로 실제 소스 장애를 명시적으로 실패시킨다. 정상 응답의 빈 결과는 장애로 오판하지 않는다.

## 3. 백테스트·API·캐시 결함 수정

대상 파일:

- `backtester.py`
- `app.py`
- `stock_db.py`

구현:

1. 총수익률과 낙폭 계산의 기준 자산을 첫 거래 후 평가액이 아니라 `initial_capital`로 고정한다.
2. 매수 로트를 FIFO로 부분/전량 청산하고 매수 수수료와 매도 비용을 로트별로 비례 배분한다.
3. 거래 상세 금액에서 체결가에 이미 포함된 슬리피지를 중복 차감하지 않는다.
4. DB 뷰어 통계 키를 백엔드 응답과 맞춘다.
5. 잘못된 테이블·페이지·백테스트 인수를 400/404 JSON으로 반환한다.
6. 데이터 갱신과 백테스트 상태를 작업 스레드 시작 전에 원자적으로 `loading`으로 바꾼다.
7. 티커 맵 캐시는 가장 최근 갱신 시각으로 판단하고, 지수 데이터 범위가 이미 있으면 API 호출을 생략한다.
8. 새 DuckDB는 `ticker_map.json`으로 초기화하고, KRX 인증 또는 호출 실패 시 주가와 KOSPI 지수를 yfinance로 보완한다.

## 4. 실행 환경과 문서 정리

대상 파일:

- `requirements.txt`
- `.github/workflows/daily_report.yml`
- `.gitignore`
- `README.md`

구현:

1. 사용하지 않게 된 Selenium, webdriver-manager, BeautifulSoup 의존성과 Chrome 설치 단계를 제거한다.
2. GitHub Actions에서 표준 라이브러리 회귀 테스트를 먼저 실행한다.
3. 생성 리포트 파일을 Git 추적 대상에서 제외한다.
4. README의 데이터 소스, 설치 방식, 테스트 명령, 자동 실행 조건, 프로젝트 구조를 실제 구현에 맞춘다.

## 5. 검증과 전달

실행 순서:

1. 새 테스트가 기존 코드에서 기대대로 실패하는지 확인한다.
2. 수정 후 `python -m unittest discover -s tests -v`를 실행한다.
3. `ruff check`, `python -m py_compile`, 모듈 import 스모크를 실행한다.
4. 실데이터로 세 소스 수집과 정적 리포트/일일 리포트 핵심 경로를 스모크 테스트한다.
5. `git diff --check`, 변경 파일 리뷰, 비밀정보 스캔, `git status`를 확인한다.
6. Lore 프로토콜 형식으로 커밋하고 현재 추적 브랜치 `origin/master`에 푸시한다.

중단 조건: 위 검증이 통과하고 원격 브랜치가 새 커밋을 가리키는 것이 확인되면 완료한다.
