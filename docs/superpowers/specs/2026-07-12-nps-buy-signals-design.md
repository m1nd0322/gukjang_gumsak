# 국민연금 신규·추가매수 신호 설계

작성일: 2026-07-12

## 목적

현재의 `국민연금 보유` 점수는 보유 기간과 매수 방향에 관계없이 모든 국민연금 보유 종목에 계속 1점을 준다. 이를 최근 국민연금 매수 행동을 나타내는 시한부 신호로 바꾼다.

- 국민연금 신규매수는 이벤트 발생일부터 3개월 동안 1점을 준다.
- 국민연금 추가매수는 해당 이벤트 발생일부터 3개월 동안 1점을 준다.
- 추가매수가 다시 발생하면 3개월 유효기간을 새 이벤트 날짜부터 다시 계산한다.
- 3개월이 되는 날에는 카테고리에서 제거하고 기존 1점을 회수한다.
- 국민연금 점수는 여러 매수 이벤트가 있어도 종목당 최대 1점이다.
- 기존 세 가지 점수 체계를 유지하므로 종합점수 최댓값은 3점이다.

여기서 사용자가 말한 `-1 스코어`는 음수 카테고리를 새로 만드는 것이 아니라, 만료된 국민연금 매수 신호가 기여하던 1점을 종합점수에서 제거한다는 의미로 정의한다.

## 확인된 원천 데이터

### 탐지 범위

이 기능이 탐지하는 것은 FnGuide 공개 화면에 국민연금공단 주요주주 변동으로 나타나는 이벤트다. 공개 화면에 나타나지 않는 공시 기준 미만의 소규모 매매까지 국민연금의 전체 주문 내역처럼 해석하지 않는다. 따라서 `신규매수`는 국민연금이 해당 종목을 처음 한 주 매수한 시점이 아니라 다음 중 확인 가능한 시점을 뜻한다.

- FnGuide에 신규보고 또는 신규주요주주로 공개된 시점
- 이 애플리케이션이 완전한 이전 Snapshot과 비교해 국민연금 보유의 신규 등장을 처음 확인한 시점

이 제한은 대시보드와 README에도 표시한다.

### Snapshot

`https://wcomp.fnguide.com/CompanyInfo/Snapshot?cmp_cd=<종목코드>`의 국민연금공단 행은 다음 현재값을 제공한다.

- 보통주 보유량
- 현재 지분율
- 최종변동일

이 화면만으로는 최종변동이 매수인지 매도인지 알 수 없다.

### ShareAnalysis

`https://wcomp.fnguide.com/CompanyInfo/ShareAnalysis?cmp_cd=<종목코드>`의 `주주변동내역`은 최근 20건에 대해 다음 값을 제공한다.

- 대표주주와 변동주주
- 변동일
- 주요변동사유
- 주식종류
- 변동 전 주식수
- 증감 주식수
- 변동 후 주식수
- 변동 후 지분율

이 데이터로 국민연금의 양수·음수 변동과 신규보고를 구분할 수 있다. 다만 종목 전체 주주의 최근 20건만 제공하므로, 3개월이 지나기 전에 국민연금 이벤트가 화면에서 밀려날 수 있다. 따라서 계산 때마다 화면만 다시 읽는 방식으로는 정확한 3개월 유지가 보장되지 않는다.

## 검토한 접근법

### 1. ShareAnalysis 최근 20건만 매번 계산

구현과 운영이 가장 단순하고 별도 상태가 필요 없다. 그러나 다른 주주의 변동이 많은 종목에서는 국민연금 이벤트가 20건 밖으로 밀려 +1점이 3개월보다 일찍 사라진다. 핵심 만료 계약을 지키지 못하므로 채택하지 않는다.

### 2. Snapshot 변경만 이전 실행과 비교

신규 등장과 보유주식 수 증가를 안정적으로 찾을 수 있다. 하지만 첫 실행에는 비교 기준이 없어서 최근 3개월 안에 이미 발생한 매수 신호를 복원할 수 없고, 변동 사유도 표시할 수 없다. 단독 방식으로 채택하지 않는다.

### 3. ShareAnalysis 이벤트와 영속 상태 결합

ShareAnalysis로 원천 이벤트를 판별하고, 한 번 발견한 유효 이벤트를 상태 파일에 보존한다. Snapshot의 이전·현재 보유량 비교를 보조 판정으로 사용한다. 최근 20건 제한을 넘어서도 정확한 만료일까지 신호를 유지하고, 첫 실행도 가능한 범위에서 복원할 수 있어 이 방식을 채택한다.

## 판정 규칙

### 현재 보유 조건

점수 대상은 현재 Snapshot에 국민연금공단 행이 있는 종목으로 제한한다. 이전에 매수 이벤트가 있었더라도 현재 국민연금 보유 목록에서 사라진 종목은 활성 신호와 점수에서 제거한다.

### 신규매수

다음 중 하나를 만족하는 양수 보유 이벤트를 `신규매수`로 분류한다.

1. ShareAnalysis 변동 사유가 `신규보고(+)` 또는 `신규주요주주(+)`처럼 `신규`로 시작한다.
2. 변동 전 보통주가 0이고 변동 후 보통주가 양수이다.
3. 이전의 완전한 Snapshot에는 없었지만 새 Snapshot에 국민연금 보유 행이 나타난다.

최초 상태 파일을 만드는 실행에서는 현재 보유 종목 전체를 신규매수로 간주하지 않는다. ShareAnalysis에서 확인되는 최근 3개월의 적격 양수 이벤트만 초기 활성 신호로 복원하고, 나머지는 비교용 기준선으로만 저장한다.

### 추가매수

이미 국민연금 보유 기준선이 있는 종목에서 다음 중 하나를 만족하면 `추가매수`로 분류한다.

1. ShareAnalysis의 국민연금 보통주 이벤트에서 `증감 > 0`이고 `변동 후 > 변동 전`이다.
2. 완전한 이전 Snapshot과 새 Snapshot을 비교했을 때 보통주 보유량이 증가했고 새 `최종변동일`이 이전보다 늦다.

ShareAnalysis의 원문 변동 사유는 결과에 함께 보존한다. 주식 수 증가가 없는 지분율 변화만으로는 추가매수로 판정하지 않는다. 이는 자사주 소각이나 발행주식 수 변화로 지분율만 달라진 경우를 매수로 오인하지 않기 위한 경계다.

### 매도와 감소

`증감 <= 0` 또는 새 Snapshot의 보통주 보유량 감소는 새 점수를 만들거나 유효기간을 갱신하지 않는다. 단, 현재 보유 행이 완전히 사라지면 기존 활성 신호를 즉시 제거한다.

### 중복과 우선순위

- 같은 종목의 여러 적격 매수 이벤트 중 날짜가 가장 늦은 이벤트를 사용한다.
- 같은 날짜에 신규매수와 추가매수가 함께 있으면 신규매수를 우선 표시한다.
- 새 추가매수는 기존 이벤트 종류와 관계없이 이벤트 날짜와 만료일을 갱신한다.
- 점수는 이벤트 개수와 관계없이 1점이다.

## 3개월 유효기간

달력 기준 3개월을 사용한다.

```text
event_date <= as_of_date < event_date + 3 calendar months
```

예를 들어 2026-07-12 이벤트는 2026-10-11까지 활성이고 2026-10-12부터 만료된다. 월말 날짜는 대상 월의 마지막 날로 보정한다. 예를 들어 2026-01-31의 3개월 후는 2026-04-30이다.

각 활성 행에는 다음 값을 노출한다.

- `매수구분`: `신규매수` 또는 `추가매수`
- `매수일`: `YYYY-MM-DD`
- `만료일`: 점수에서 빠지는 첫 날짜
- `변동사유`
- `변동전`, `증감`, `변동후`, `지분율(%)`

## 상태 파일

저장소 루트의 `nps_state.json`을 사용하고 Git에는 포함하지 않는다. 스키마는 버전 필드를 둔다.

```json
{
  "version": 1,
  "updated_at": "2026-07-12T08:00:00+09:00",
  "holdings": {
    "005930": {
      "종목명": "삼성전자",
      "보통주": 458637667,
      "지분율": 7.84,
      "최종변동일": "2022-08-16"
    }
  },
  "signals": {
    "069620": {
      "종목명": "대웅제약",
      "매수구분": "신규매수",
      "매수일": "2026-07-06",
      "만료일": "2026-10-06",
      "변동사유": "신규주요주주(+)",
      "변동전": 1153635,
      "증감": 16881,
      "변동후": 1170516,
      "지분율": 10.10
    }
  }
}
```

상태 갱신 규칙은 다음과 같다.

1. Snapshot과 ShareAnalysis 수집·검증이 끝나기 전에는 기존 파일을 변경하지 않는다.
2. 임시 파일에 완전한 JSON을 쓴 뒤 `os.replace`로 원자적으로 교체한다.
3. 손상되거나 지원하지 않는 버전의 상태는 빈 상태로 조용히 덮지 않고 오류로 처리한다.
4. 만료 신호와 현재 보유에서 사라진 신호는 다음 성공적인 저장에서 제거한다.
5. 시간대는 `Asia/Seoul`, 판정 단위는 날짜로 고정한다.

## 수집 및 데이터 흐름

1. 기존 JSON 피드에서 턴어라운드와 순매수전환을 수집한다.
2. Snapshot 전 종목 조회로 현재 국민연금 보유 행과 종목코드를 수집한다.
3. 현재 국민연금 보유 종목의 ShareAnalysis를 제한된 병렬 요청으로 조회한다.
4. ShareAnalysis 이벤트, 이전 상태, Snapshot 차이를 결합해 최신 적격 매수 이벤트를 판정한다.
5. 현재 날짜에 활성인 국민연금 매수 신호만 세 번째 점수 데이터셋으로 반환한다.
6. 세 데이터셋을 기존 `calculate_scores`에 전달한다.
7. 완전한 판정 결과를 `nps_state.json`에 원자적으로 저장한다.

`screening.py`가 웹 대시보드, 정적 HTML 리포트, 텔레그램 일일 리포트의 공통 데이터 계층이라는 기존 구조를 유지한다. 이벤트 파싱과 상태 전이는 별도 `nps_tracker.py` 모듈로 분리해 네트워크 수집과 순수 판정 로직을 독립 테스트할 수 있게 한다.

## 화면과 리포트

- 점수 출처 이름을 `국민연금`에서 `국민연금 신규/추가매수`로 변경한다.
- 대시보드의 세 번째 카테고리와 통계를 `국민연금 신규/추가매수`로 표시한다.
- 정적 HTML의 태그와 세부 정보에 매수구분·매수일·만료일을 표시한다.
- 텔레그램 통계와 상위 종목 출처도 같은 이름을 사용한다.
- 내부 호환성이 필요한 기존 통계 키 `nps_count`는 유지하되 의미를 활성 국민연금 매수 신호 수로 바꾼다.

## GitHub Actions 상태 지속

GitHub Actions는 실행마다 새 파일시스템을 사용하므로 일일 리포트 작업에 `nps_state.json` 캐시 복원·저장 단계를 추가한다.

- 실행 전 가장 최근의 `nps-state-` 캐시를 복원한다.
- 실행마다 고유한 키로 갱신된 상태를 저장한다.
- 캐시가 없으면 ShareAnalysis 최근 20건을 이용한 초기화 규칙을 적용한다.
- 캐시 저장 실패는 리포트 결과를 거짓 성공으로 만들지 않도록 워크플로 실패로 남긴다.

## 오류 처리

- Snapshot의 기존 80% 유효 페이지 검증을 유지한다.
- 첫 상태 초기화에서 ShareAnalysis 유효 페이지 비율이 80% 미만이면 국민연금 소스 전체를 실패시킨다.
- 기존 상태가 있을 때 일부 ShareAnalysis 조회가 실패하면 이전 신호와 Snapshot 차이 판정을 사용하되 실패 수를 경고한다.
- 모든 소스가 필수인 웹 갱신과 일일 리포트에서는 원천 검증 실패 시 새 상태와 점수를 게시하지 않는다.
- 웹 앱은 기존 정상 캐시를 유지하고 오류 메시지를 표시한다.

## 테스트 계약

### 파서

- 국민연금 행만 추출하고 다른 기관 행은 제외한다.
- 신규보고, 신규주요주주, 양수 추가매수, 음수 매도를 구분한다.
- 쉼표가 포함된 주식 수와 날짜를 정규화한다.
- 요청 종목코드와 실제 페이지 종목코드가 다르면 거부한다.

### 상태 전이

- 최초 실행은 모든 기존 보유 종목을 신규매수로 만들지 않는다.
- 새 국민연금 보유 행은 신규매수 신호를 만든다.
- 보유주식 수 증가와 더 최신의 변동일은 추가매수 신호를 만든다.
- 감소는 유효기간을 갱신하지 않는다.
- 추가매수는 기존 만료일을 새 매수일 기준으로 갱신한다.
- 같은 종목의 여러 이벤트는 1점만 만든다.
- 현재 보유 행이 사라지면 신호를 제거한다.
- 이벤트 날짜 당일과 만료 전날은 활성, 만료일은 비활성이다.
- 월말 3개월 계산을 검증한다.
- 손상된 상태와 부분 수집은 기존 상태를 덮어쓰지 않는다.

### 통합

- 국민연금 매수 신호가 있는 종목은 세 번째 기준으로 1점을 받는다.
- 만료 후 같은 입력에서 종합점수가 정확히 1점 낮아지고 출처에서 국민연금 카테고리가 사라진다.
- 턴어라운드와 순매수전환 점수는 영향을 받지 않는다.
- 대시보드 API, 정적 HTML, 텔레그램 메시지가 매수구분·매수일·만료일을 노출한다.
- GitHub Actions가 상태 파일을 복원하고 다음 실행용 캐시를 저장한다.

## 마이그레이션과 호환성

- 기존 `cache_data.json`의 모든 국민연금 보유 행은 새 점수의 기준으로 재사용하지 않는다.
- 첫 성공적인 새 수집에서 `nps_state.json`을 만들고 활성 신호 기준으로 결과 캐시를 교체한다.
- 이전에 장기 보유만으로 받던 국민연금 점수는 의도적으로 사라진다.
- 결과 행의 `종합점수`, `출처`, `순위` 계약과 2점 이상 백테스트 선택 규칙은 유지한다.

## 완료 조건

1. 신규매수 종목이 이벤트 날짜부터 3개월 동안 정확히 1점을 받는다.
2. 만료일에 해당 1점과 카테고리가 제거된다.
3. 추가매수 이벤트가 새 1점을 중복 적립하지 않고 유효기간을 새 날짜부터 갱신한다.
4. 매도·감소 이벤트가 유효기간을 갱신하지 않는다.
5. 대시보드, 정적 리포트, 일일 텔레그램 리포트가 동일한 공통 판정을 사용한다.
6. 로컬 재시작과 GitHub Actions 재실행 사이에서도 활성 신호가 유지된다.
7. 단위·통합·회귀 테스트와 정적 검사가 모두 통과한다.

첫 상태 초기화 시점보다 앞서 발생했고 이미 ShareAnalysis 최근 20건 밖으로 밀려난 이벤트는 소급 복원할 수 없다. 최초 성공 실행 이후에는 매일 상태와 Snapshot 차이를 보존해 같은 누락이 발생하지 않게 하는 것을 운영상 완료 기준으로 삼는다.
