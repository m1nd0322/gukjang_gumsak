# 변동성 가중·트레일링 스탑·스탑로스 전략 설계

## 목적

기존 변동성 가중·트레일링 스탑 전략에 매수가 기준 고정 손절을 결합한 별도 웹 백테스트 전략을 제공한다. 사용자는 스탑로스 비율을 직접 지정할 수 있고, 기존 전략의 결과와 API 호환성은 유지한다.

## 현재 상태

- `run_volatility_trailing_stop()`은 역변동성 비중으로 매수한다.
- 보유 중 최고 종가 대비 `-10%` 이하로 하락하면 전량 매도한다.
- 매도 후 5거래일 동안 재진입하지 않고 이후 재진입을 허용한다.
- 매수가 기준 고정 손절은 없다.
- 웹 백테스트는 5개 전략을 제공한다.
- 2026-07-14 20:48:10 KST 스크리닝 결과에는 `국민연금 신규/추가매수`만으로 종합점수 1점인 종목이 84개 있다. 실제 검증에서는 이 숫자를 하드코딩하지 않고 실행 시점의 동일 조건 후보 수를 다시 계산한다.

## 범위

- `BacktestEngine`이 선택적으로 고정 스탑로스를 적용할 수 있게 한다.
- 웹 백테스트에 새 전략 `vol_trailing_stop_loss`를 추가한다.
- 새 전략의 스탑로스 기본값을 `7%`로 설정한다.
- 웹과 API에서 스탑로스 비율을 커스텀할 수 있게 한다.
- 실행 결과 설정에 적용 스탑로스 비율을 기록한다.
- 국민연금 매수만으로 종합점수 1점인 현재 종목 집합으로 실제 종단 간 백테스트를 검증한다.
- README의 전략 수, 전략 설명, 커스텀 설정을 갱신한다.

## 전략 계약

### 전략 식별자와 표시명

| 구분 | 값 |
| --- | --- |
| API 전략 키 | `vol_trailing_stop_loss` |
| 화면 표시명 | 변동성 가중 + 트레일링 스탑 + 스탑로스 |
| 기본 스탑로스 | `7%` |
| 허용 범위 | `0.1%` 이상 `50%` 이하 |
| 입력 단위 | `0.5%` |
| 트레일링 스탑 | 보유 중 최고 종가 대비 `-10%` |
| 쿨다운 | 전량 매도 후 5거래일 |
| 재진입 | 쿨다운 종료 후 허용 |

### 기준 가격

- 트레일링 스탑은 기존과 같이 보유 기간의 최고 종가를 기준으로 한다.
- 스탑로스는 `Portfolio.positions[ticker]['avg_price']`의 실제 평균 체결가를 기준으로 한다.
- 평균 체결가에는 매수 슬리피지가 반영되어 있으므로 실제 포트폴리오 원가와 일치한다.
- 위험 조건은 일별 종가로 평가한다. 장중 고가·저가를 사용하지 않는다.

### 매도 조건

보유 종목마다 최고 종가를 갱신한 뒤 다음 두 조건을 계산한다.

```text
트레일링 수익률 = (현재 종가 / 보유 중 최고 종가 - 1) × 100
매수가 수익률   = (현재 종가 / 실제 평균 체결가 - 1) × 100

매도 = 트레일링 수익률 <= -10
    OR 매수가 수익률 <= -사용자 스탑로스 비율
```

- 두 조건 중 하나라도 충족하면 해당 종목을 한 번만 전량 매도한다.
- 같은 날 두 조건이 동시에 충족해도 중복 매도하지 않는다.
- 매도 뒤에는 기존과 같은 쿨다운과 재진입 규칙을 적용한다.
- 가격이나 실제 포지션이 없으면 해당 일자의 위험 조건을 건너뛴다.

### 기존 전략 호환성

기존 `run_volatility_trailing_stop()`에 선택적 `stop_loss_pct` 인자를 추가한다. 기본값은 `None`이며, 기존 `vol_trailing_stop` 호출은 이 값을 전달하지 않아 이전과 똑같이 트레일링 스탑만 적용한다. 새 `vol_trailing_stop_loss` 전략만 양수 비율을 전달한다.

복제된 전략 루프를 새로 만들지 않는다. 변동성 계산, 매수, 트레일링 스탑, 쿨다운, 재진입 로직을 한 구현에서 공유해 두 전략이 손절 활성화 여부만 다르게 한다.

## 웹과 API 계약

### 요청

`POST /api/backtest/run`은 기존 본문에 선택적 `stop_loss` 숫자 필드를 받는다.

```json
{
  "period": 6,
  "capital": 100000000,
  "strategy": "vol_trailing_stop_loss",
  "stop_loss": 7,
  "scores": [1],
  "items": ["nps"]
}
```

- `stop_loss`를 생략하면 `7.0`으로 처리한다.
- 문자열로 숫자를 전달하면 기존 숫자 설정과 같은 방식으로 실수 변환한다.
- 유한한 숫자 `0.1` 이상 `50.0` 이하만 허용한다.
- 범위를 벗어나거나 숫자로 변환할 수 없으면 백그라운드 작업을 시작하지 않고 HTTP 400을 반환한다.
- 새 전략이 아닌 요청에서도 필드가 제공되면 같은 형식 검증을 수행하지만 전략 실행에는 적용하지 않는다.

### 작업 전달과 결과

- 기존 점수·항목 필터의 스레드 위치 인자를 유지하고 `stop_loss_pct`는 키워드 인자로 작업에 전달한다.
- 새 전략은 `lookback=20`, 트레일링 `stop_pct=-10.0`, `cooldown=5`, `reentry=True`, 사용자 `stop_loss_pct`로 엔진을 호출한다.
- 결과의 `config`에 `stop_loss_pct`를 기록한다. 새 전략이 아니어도 요청에 사용된 설정값을 기록해 요청 재현성을 유지한다.

### 화면

- 전략 선택 목록에 새 전략을 추가한다.
- 설정 영역에 `스탑로스 (%)` 숫자 입력을 추가한다.
- 기본값은 `7`, `min=0.1`, `max=50`, `step=0.5`다.
- 라벨에 새 스탑로스 전략에서만 적용된다는 안내를 표시한다.
- 실행 요청의 JSON에 `stop_loss`를 숫자로 포함한다.
- 기존 전략 선택과 필터 UI는 변경하지 않는다.

## 실행 흐름

```text
백테스트 화면
  -> 새 전략 + 스탑로스 비율 + 점수/항목 필터 전송
  -> API 숫자·범위·전략 검증
  -> 백그라운드 작업에 stop_loss_pct 키워드 전달
  -> 점수/항목 후보 필터 적용
  -> 가격 데이터 적재
  -> 역변동성 가중 매수
  -> 매일 트레일링 스탑 OR 매수가 스탑로스 평가
  -> 매도·쿨다운·재진입
  -> 결과 config에 전략과 스탑로스 기록
```

## 오류 처리

- 스탑로스 입력 형식이나 범위 오류는 HTTP 400으로 반환한다.
- 대상 종목 없음, 티커 매핑 실패, 가격 데이터 없음은 기존 백테스트 오류 흐름을 유지한다.
- 실제 검증 중 일부 종목의 티커 또는 가격을 찾지 못하면 `unmatched`와 적재 종목 수를 확인한다. 모든 필터 후보가 매핑 단계에 투입됐는지 `total_stocks + len(unmatched)`로 대조한다.
- 외부 가격 데이터 장애로 종단 간 검증이 완료되지 않으면 커밋·푸시 전에 원인을 해결하거나 검증 실패로 보고한다.

## 검증 전략

### 엔진 단위 검증

결정적 가격 시계열과 거래비용 0인 단일 종목을 사용한다.

1. `100 → 94 → 93`에서 기본 7% 스탑로스가 93에 전량 매도하는지 확인한다. 최고가 대비 10% 트레일링 조건은 충족하지 않으므로 스탑로스만 검증된다.
2. 사용자 비율 5%와 8%를 각각 적용해 매도 시점이 달라지는지 확인한다.
3. `100 → 120 → 108`에서 매수가 대비 수익 상태인 채 최고가 대비 10% 하락하여 트레일링 스탑이 작동하는지 확인한다.
4. `stop_loss_pct=None`인 기존 전략은 `100 → 93`에서 매도하지 않아 이전 동작을 유지하는지 확인한다.
5. 스탑로스 매도 뒤 5거래일 동안 재진입하지 않고 그 이후 재진입하는지 확인한다.

### API·화면 검증

- 새 전략 키를 허용하고 엔진 메서드에 `stop_loss_pct=7.0`을 전달하는지 확인한다.
- 커스텀 값이 손실 없이 전달되고 결과 설정에 기록되는지 확인한다.
- 비숫자, NaN, 무한대, `0.1` 미만, `50` 초과 입력이 HTTP 400이고 작업을 시작하지 않는지 확인한다.
- 화면에 새 전략과 스탑로스 입력 속성이 있고 JSON 요청에 값이 포함되는지 확인한다.

### 국민연금 단독 1점 실제 검증

실행 시점의 스크리닝 결과에서 다음 정확 조건으로 후보를 다시 계산한다.

```text
종합점수 == 1
AND 출처 == 국민연금 신규/추가매수
```

웹 API에 `scores: [1]`, `items: ["nps"]`, 새 전략, `stop_loss: 7`, 기간 6개월을 전송한다. 완료 뒤 다음을 확인한다.

- API 상태가 `done`이다.
- 결과 전략 키가 `vol_trailing_stop_loss`다.
- 결과 스탑로스가 `7.0`이다.
- `total_stocks + len(unmatched)`가 실행 직전 후보 수와 같다.
- `loaded_stocks`가 1개 이상이다.
- 메트릭, 비용 요약, 종목별 성과, 거래 내역이 직렬화 가능한 결과로 반환된다.
- 필터 결과가 국민연금 단독 1점 외 종목을 포함하지 않는다.
