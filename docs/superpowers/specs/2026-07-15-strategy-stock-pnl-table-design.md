# 전략 종목별 손익 표 설계

## 목적

백테스트 결과의 `종목별 성과` 표가 전략의 실제 매매 결과가 아니라 각 종목의 백테스트 기간 첫 종가와 마지막 종가를 비교하고 있어, 스탑로스 등 전략이 적용된 손익으로 오해될 수 있다. 원가격 성과 표를 제거하고 거래 비용과 보유 상태를 반영한 `전략 종목별 손익` 표로 교체한다.

## 현재 상태와 문제

- `BacktestEngine._calc_stock_performance()`은 가격 데이터가 적재된 모든 종목의 첫 종가, 마지막 종가, 단순 수익률, 원가격 MDD를 계산한다.
- 이 계산은 실제 매수 여부, 매도 시점, 스탑로스, 트레일링 스탑, 재진입, 보유 수량, 슬리피지, 수수료와 세금을 보지 않는다.
- 웹은 결과의 `stock_performance`를 `종목별 성과` 표로 표시한다.
- 일일 보고서도 같은 값을 `개별 종목 수익률`로 표시한다.
- 따라서 포트폴리오가 손절로 손실을 제한해도 표에는 이후 원가격 하락분까지 포함된 더 큰 손실이 나타날 수 있다.

## 결정

백엔드가 거래 기록을 기준으로 종목별 전략 손익을 한 번 집계하고, 웹과 일일 보고서가 같은 결과를 사용한다.

- 기존 `stock_performance` 결과 필드를 제거한다.
- 새 결과 필드 이름은 `strategy_stock_performance`로 한다.
- `_calc_stock_performance()`을 제거하고 `_calc_strategy_stock_performance()`을 추가한다.
- 가격만 적재되고 거래가 한 번도 발생하지 않은 종목은 결과에 포함하지 않는다.
- 기존 `trades`와 `trades_by_stock` 결과는 상세 이력과 기존 소비자를 위해 유지한다.

프런트엔드가 `trades`를 다시 집계하는 방식은 웹과 일일 보고서에 계산이 중복되고 API 자체의 의미도 고쳐지지 않으므로 사용하지 않는다. 기존 `stock_performance`의 의미만 조용히 바꾸는 방식도 동일한 필드가 전혀 다른 계약을 갖게 되므로 사용하지 않는다.

## 손익 계산 계약

### 종목별 누적 총매입금액

사용자가 선택한 기준에 따라 종목별 전략 손익률의 분모는 해당 종목 모든 거래 로트의 누적 총매입금액이다.

```text
로트 총매입금액 = 실제 매수 체결가 × 수량 + 매수 수수료
종목 누적 총매입금액 = 모든 청산·보유 로트의 로트 총매입금액 합계
```

- 실제 매수 체결가에는 매수 슬리피지가 이미 반영되어 있다.
- 부분 청산으로 하나의 매수 로트가 청산분과 보유분으로 나뉘면, 엔진이 배분한 수량과 매수 수수료를 각각 합산한다. 분할 전 총매입원가는 보존된다.
- 재진입한 거래도 새로운 로트로 누적한다. 따라서 손익률은 현재 투입금뿐 아니라 백테스트 동안 해당 종목에 투입한 누적 원금 대비 성과다.

### 실현손익

```text
종목 실현손익 = 상태가 closed인 모든 로트의 TradeRecord.pnl 합계
```

`TradeRecord.pnl`은 다음 항목을 모두 반영한 순손익이다.

- 매수 슬리피지
- 매수 수수료
- 매도 슬리피지
- 매도 수수료
- 증권거래세

### 미실현손익

```text
보유 로트 평가금액 = 백테스트 마지막 종가 × 보유 수량
보유 로트 미실현손익 = 평가금액 - 로트 총매입금액
종목 미실현손익 = 상태가 open인 모든 로트의 미실현손익 합계
```

- 평가가격은 해당 종목 가격 데이터의 마지막 종가를 사용한다.
- 매수 슬리피지와 매수 수수료는 총매입금액에 포함된다.
- 아직 매도하지 않았으므로 가상의 매도 슬리피지, 매도 수수료와 세금은 차감하지 않는다. 이는 엔진의 마지막 포트폴리오 평가액 계산과 같은 기준이다.
- 열린 거래인데 마지막 가격이 없으면 `0`으로 조용히 계산하지 않고 종목코드를 포함한 `ValueError`를 발생시킨다. 정상 백테스트에서는 거래를 생성한 종목에 가격 데이터가 존재하므로 이 오류는 결과 왜곡을 막는 불변식 검사다.

### 총손익과 손익률

```text
종목 총손익 = 종목 실현손익 + 종목 미실현손익

종목 전략 손익률(%) =
    종목 총손익 / 종목 누적 총매입금액 × 100
```

- 누적 총매입금액이 `0`이면 손익률은 `0.0`으로 처리해 0으로 나누지 않는다.
- 계산은 원본 실수값으로 수행하고, 응답 직전에 금액은 원 단위, 손익률은 소수점 둘째 자리로 반올림한다.
- 종목별 반올림 때문에 표시 금액 합계와 전체 포트폴리오 손익 사이에는 최대 수 원의 차이가 생길 수 있으므로 원본 계산 기준으로 일치 여부를 검증한다.

## 결과 스키마

`strategy_stock_performance`는 총손익 내림차순 배열이다. 각 행은 다음 필드를 제공한다.

| 필드 | 의미 |
| --- | --- |
| `ticker` | 종목코드 |
| `name` | 종목명 |
| `trade_count` | 청산 및 보유 거래 로트 수의 합계 |
| `closed_count` | 청산된 거래 로트 수 |
| `open_count` | 현재 보유 중인 거래 로트 수 |
| `total_buy_amount` | 매수 체결금액과 매수 수수료를 포함한 누적 총매입금액 |
| `realized_pnl` | 모든 청산 로트의 비용 차감 후 실현손익 |
| `unrealized_pnl` | 마지막 종가 기준 보유 로트의 미실현손익 |
| `total_pnl` | 실현손익과 미실현손익의 합계 |
| `return_pct` | 누적 총매입금액 대비 총손익률 |

`trade_count`는 주문 요청 횟수가 아니라 현재 `TradeRecord`가 표현하는 거래 로트 수다. 부분 청산으로 로트가 분리될 수 있으므로 화면에서는 `거래건수`로 표시한다.

## 웹 화면

기존 `종목별 성과` 표와 원가격 기반 열을 완전히 제거하고 다음 표로 교체한다.

### 제목

`전략 종목별 손익`

### 열

1. 종목명
2. 종목코드
3. 거래건수
4. 청산건수
5. 보유건수
6. 누적 총매입금액
7. 실현손익
8. 평가손익
9. 총손익
10. 손익률

- 금액은 기존 숫자 포매터로 천 단위 구분해 표시한다.
- 실현손익, 평가손익, 총손익과 손익률은 양수면 녹색, 음수면 빨간색을 적용한다.
- `0`은 양수 기호 없이 중립적으로 표시한다.
- 행 순서는 백엔드가 제공한 총손익 내림차순을 유지한다.
- 시작가, 종료가와 종목 원가격 MDD는 표시하지 않는다.
- 렌더러 이름도 원가격 의미가 남지 않도록 `renderStrategyStockTable()`로 바꾼다.

## 일일 보고서

- `stock_performance` 대신 `strategy_stock_performance`를 사용한다.
- 제목은 `전략 종목별 손익`으로 변경한다.
- 상위 10개 종목을 총손익 내림차순으로 표시한다.
- 각 행에는 종목명, 총손익과 전략 손익률을 표시한다.
- 원가격 MDD는 제거한다.
- 데이터가 없으면 해당 섹션을 생략하는 기존 흐름을 유지한다.

## 포트폴리오 손익 대조

마지막 스냅샷이 있는 정상 백테스트에서는 모든 종목의 반올림 전 총손익 합계가 다음 값과 일치해야 한다.

```text
마지막 포트폴리오 평가액 - 최초 자본
```

그 이유는 청산 로트에는 모든 진입·청산 비용이 `pnl`에 포함되고, 보유 로트에는 마지막 종가 평가액과 이미 현금에서 빠져나간 진입원가의 차이가 포함되기 때문이다. 이 불변식을 단위 테스트와 실제 백테스트 검증에 사용한다.

## 테스트 전략

### 엔진 단위 테스트

최초 자본 2,000원에서 거래비용을 명확히 확인할 수 있도록 슬리피지와 세금은 `0%`, 매수·매도 수수료는 `1%`인 결정적 시나리오를 사용한다.

1. 종목 AAA를 100원에 10주 매수한다.
   - 체결금액 1,000원, 매수 수수료 10원, 총매입금액 1,010원
2. AAA를 110원에 10주 매도한다.
   - 매도금액 1,100원, 매도 수수료 11원, 실현손익 79원
3. AAA를 다시 100원에 5주 매수하고 마지막 종가를 120원으로 평가한다.
   - 총매입금액 505원, 평가금액 600원, 미실현손익 95원
4. 결과를 확인한다.
   - 누적 총매입금액 1,515원
   - 실현손익 79원
   - 미실현손익 95원
   - 총손익 174원
   - 전략 손익률 `11.49%`
   - 거래건수 2, 청산건수 1, 보유건수 1
5. 마지막 평가액과 최초 자본의 차이가 174원인지 확인한다.
6. 가격만 적재하고 거래하지 않은 BBB가 결과에 포함되지 않는지 확인한다.
7. 여러 종목과 부분 청산에서도 종목별 총손익 합계가 전체 포트폴리오 손익과 일치하는지 확인한다.

### 웹 템플릿 테스트

- `전략 종목별 손익` 제목과 새 열이 존재하는지 확인한다.
- `strategy_stock_performance`를 렌더링하는지 확인한다.
- 기존 `종목별 성과`, `stock_performance`, 시작가, 종료가와 원가격 MDD 바인딩이 제거됐는지 확인한다.
- 손익의 양수·음수·0 표시 형식을 확인한다.

### 일일 보고서 테스트

- 새 결과 필드로 총손익과 손익률이 표시되는지 확인한다.
- 원가격 수익률과 MDD가 출력되지 않는지 확인한다.
- HTML 특수문자가 포함된 종목명이 기존처럼 이스케이프되는지 확인한다.

### 전체 검증

- 관련 엔진·웹·일일 보고서 테스트
- 전체 `unittest` 스위트
- Python `py_compile`
- Ruff
- `git diff --check`
- 새 코드로 앱을 재시작하고 `/backtest`가 HTTP 200인지 확인
- 실제 캐시 데이터를 사용한 백테스트 완료 확인
- 결과에 `strategy_stock_performance`가 있고 `stock_performance`가 없는지 확인
- 종목별 총손익 합계가 `metrics.profit_loss`와 반올림 허용오차 내에서 일치하는지 확인
- 화면에 새 표만 렌더링되고 기존 원가격 표가 없는지 확인

## 호환성과 마이그레이션

- 웹과 저장소 내 일일 보고서는 같은 변경에서 새 필드로 전환한다.
- `stock_performance`는 잘못 해석될 가능성이 큰 내부 결과 필드이므로 호환 별칭으로 남기지 않는다.
- 거래 상세 응답, CSV 내보내기, 포트폴리오 메트릭, 전략 실행 로직과 손절 조건은 변경하지 않는다.
- 외부 소비자가 `stock_performance`를 사용하고 있었다면 새 `strategy_stock_performance` 스키마로 전환해야 한다.

## 제외 범위

- 스탑로스 체결 방식이나 일별 종가 판정 방식 변경
- 종목 원가격 수익률 또는 원가격 MDD를 다른 위치에 유지
- 매도 사유 필드 추가
- 거래 상세 표나 CSV 스키마 변경
- 새로운 외부 의존성 추가
- README의 전략 설명 변경
