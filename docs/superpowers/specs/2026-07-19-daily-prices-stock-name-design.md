# 일봉 데이터 종목명 저장 설계

## 목적

`daily_prices`의 각 가격 행에 종목명을 함께 저장해 DuckDB 원본 테이블과 DB 뷰어만 보더라도 티커가 어떤 종목인지 바로 알 수 있게 한다. 기존 데이터는 `ticker_map`을 기준으로 소급 보완하고, 이후 가격 수집과 종목 매핑 갱신에서도 종목명이 자동으로 유지되게 한다.

## 현재 상태

- `daily_prices`는 `ticker`, `date`, OHLCV만 저장한다.
- 종목명은 별도 `ticker_map` 테이블에만 존재한다.
- DB 뷰어의 `daily_prices` 원본 조회에서는 종목명을 확인하려면 사용자가 직접 티커를 대조해야 한다.
- 현재 로컬 DB에는 27,700개 가격 행과 138개 티커가 있으며, 138개 티커 모두 `ticker_map`과 매핑된다.

## 대안과 결정

### 결정: `daily_prices.name`에 종목명 직접 저장

`daily_prices`에 nullable `name VARCHAR` 컬럼을 추가한다. 원본 테이블, 범용 DB 조회 API, DB 뷰어에서 별도 조인 없이 종목명을 확인하고 필터링할 수 있어 사용자의 요구를 직접 충족한다.

종목명은 티커보다 변경 가능성이 높으므로 `ticker_map`을 기준 데이터로 유지하고, 가격 행의 `name`은 조회 편의를 위한 동기화 사본으로 취급한다.

### 제외한 대안

- `daily_prices_with_name` View: 중복 저장을 피할 수 있지만 `daily_prices` 자체에는 종목명이 없어 원본 테이블의 식별 문제를 해결하지 못한다.
- DB 뷰어에서만 `ticker_map` 조인: 화면 외 SQL 소비자는 계속 티커만 보게 되고 저장 데이터 자체는 개선되지 않는다.

## 데이터 모델

`daily_prices`는 다음 컬럼을 가진다.

| 컬럼 | 타입 | 의미 |
| --- | --- | --- |
| `ticker` | `VARCHAR NOT NULL` | KRX 종목코드 |
| `date` | `DATE NOT NULL` | 거래일 |
| `open` | `DOUBLE` | 시가 |
| `high` | `DOUBLE` | 고가 |
| `low` | `DOUBLE` | 저가 |
| `close` | `DOUBLE` | 종가 |
| `volume` | `BIGINT` | 거래량 |
| `name` | `VARCHAR` | `ticker_map.name`에서 동기화한 종목명; 미매핑 시 `NULL` |

기본키 `(ticker, date)`는 그대로 유지한다. `name`은 기본키나 가격 데이터 정합성 판단에 사용하지 않는다.

## 스키마 마이그레이션

`StockDB` 초기화 시 새 DB와 기존 DB를 모두 처리한다.

1. 새 `daily_prices` 생성문에는 마지막 컬럼으로 `name VARCHAR`를 포함한다.
2. 기존 DB에는 `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`로 `name`을 추가한다.
3. `ticker_map` 생성이 보장된 뒤 `ticker`가 일치하는 기존 가격 행의 `name`을 갱신한다.
4. 이미 같은 이름이 저장된 행은 불필요하게 다시 쓰지 않는다.
5. 대응하는 `ticker_map` 행이 없으면 `name`은 `NULL`로 둔다.

마이그레이션은 앱 시작 때 반복 실행해도 결과가 달라지지 않는 멱등 동작이어야 한다. 중간 실패가 발생하더라도 다음 초기화에서 안전하게 재시도할 수 있어야 한다.

## 종목명 동기화

종목명의 기준 데이터는 `ticker_map`이다. 공용 동기화 로직은 `daily_prices.ticker = ticker_map.ticker`인 행 가운데 이름이 다르거나 비어 있는 행만 갱신한다.

동기화는 다음 시점에 실행한다.

- `StockDB` 초기화 후 기존 데이터 마이그레이션
- 저장소의 `ticker_map.json`을 DB에 적재한 직후
- KRX에서 `ticker_map`을 새로 받아 업서트한 직후

따라서 가격이 먼저 저장되어 `name`이 `NULL`이어도 이후 매핑이 들어오면 자동으로 보완된다. 상장사명이 변경되어 `ticker_map.name`이 바뀐 경우에는 같은 티커의 과거 가격 행도 최신 종목명으로 동기화한다.

매핑 업서트와 가격 행 동기화는 같은 DB 연결의 한 트랜잭션에서 처리한다. 동기화가 실패하면 매핑 변경도 롤백해 두 테이블이 서로 다른 이름을 성공 상태로 남기지 않게 한다.

## 가격 저장 흐름

`save_prices(ticker, data)`는 기존 호출 계약을 유지한다.

1. 같은 연결에서 `ticker_map`으로 티커의 현재 종목명을 조회한다.
2. 신규 가격 행에 `ticker`, `name`, 날짜, OHLCV를 함께 삽입한다.
3. `(ticker, date)` 충돌 시 OHLCV와 확인된 종목명을 갱신한다.
4. 매핑을 찾지 못한 업서트에서는 기존에 저장된 비어 있지 않은 종목명을 `NULL`로 되돌리지 않는다.

수집기와 백테스트 호출부에는 종목명 인수를 새로 요구하지 않는다. 종목명 해석과 저장 책임은 `StockDB` 내부에 둔다.

## 조회 호환성

- 범용 `get_table_schema()`와 `query_table()`은 실제 스키마를 읽으므로 별도 API 변경 없이 `name` 컬럼을 반환하고 필터링할 수 있다.
- `get_prices()`의 반환 형식은 기존처럼 날짜와 OHLCV만 유지한다. 백테스트 엔진의 입력 계약에는 종목명을 추가하지 않는다.
- `get_ticker_summary()`의 기존 `ticker_map` 조인은 유지한다. 이 API는 이미 종목명을 정상적으로 제공하므로 새 저장 컬럼 때문에 반환 계약을 바꾸지 않는다.
- 신규 인덱스는 추가하지 않는다. 현재 데이터 규모와 부분 문자열 필터 특성상 이름 인덱스의 이점이 명확하지 않으며, 기존 `ticker_map.name` 인덱스는 유지한다.

## 오류 및 일관성

- 미매핑 티커는 오류로 처리하지 않고 `name = NULL`로 저장한다.
- 종목명 조회 실패가 가격 저장 자체를 막지 않는다.
- 매핑이 없는 업서트가 기존 종목명을 지우지 않게 한다.
- 종목명 동기화 실패는 DB 초기화 또는 매핑 갱신 실패로 노출해 부분적으로 오래된 이름을 성공 상태로 숨기지 않는다.
- 새 의존성은 추가하지 않고 기존 DuckDB 연결과 SQL만 사용한다.

## 검증

- 새 DB의 `daily_prices`에 `name VARCHAR`가 생성된다.
- 구형 스키마 DB를 초기화하면 컬럼이 추가되고 기존 행이 `ticker_map`으로 보완된다.
- 매핑된 티커의 신규 가격 저장과 재업서트에 종목명이 포함된다.
- 미매핑 티커는 `NULL`로 저장되고, 이후 매핑 적재 시 이름이 채워진다.
- 종목명 변경 후 매핑 갱신을 실행하면 기존 가격 행의 이름도 변경된다.
- 매핑이 일시적으로 없더라도 기존의 비어 있지 않은 이름은 가격 업서트로 지워지지 않는다.
- DB 뷰어용 스키마·조회에서 `name`을 반환하고 종목명 필터가 동작한다.
- 기존 `get_prices()` 계약과 백테스트 회귀 테스트가 유지된다.
- 전체 단위 테스트, Ruff, Python 컴파일 검사를 통과한다.
- 실제 `stock_data.duckdb` 마이그레이션 후 매핑 가능한 행 중 `name IS NULL`인 행이 0건인지 확인한다.

## 문서 및 배포

- README의 `daily_prices` 설명을 종목코드·종목명·일봉 OHLCV 저장으로 갱신한다.
- 코드와 테스트가 통과한 뒤 실제 로컬 DuckDB를 `StockDB` 초기화 경로로 한 번 마이그레이션한다.
- Git에서 제외된 실제 DuckDB 파일은 커밋 대상에 포함하지 않는다.

## 제외 범위

- 종목명 변경 이력 보존
- `ticker_map` 제거 또는 정규화 구조 폐기
- `get_prices()` 응답에 종목명 추가
- 별도 종목명 검색 API나 신규 화면 추가
- 미매핑 티커 이름을 외부 API에서 즉시 강제 조회
